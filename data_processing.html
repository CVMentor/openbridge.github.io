Openbridge Data Bridge Services

Overview

Openbridge Event Processing Service is a distributed real-time computation system for processing fast, large streams of data. Processing services are called “topologies”. A topology is a network of resources needed to perform a collection of processes and computations on a “event”. An event is an ordered set of values and typically equates to a “record” in a RDBMS.

Topology

A topology contains processing logic and algorithms to perform increasingly complex synchronous or asynchronous operations on events flowing through the system. As a result, topologies encapsulate all the relevant “intelligence” on how to process data. Topologies are extremely flexible, allowing customers to deploy complex data processing routines based on their business requirements as data is streamed through the system. Events: Typically an event can be equated with a record. The system guarantees every event will be fully processed. This provide an “at-least-once” processing guarantee. Messages are only replayed when there are failures. Combined with Data Replay Services the system is easily able to reprocess current or historical data. There are two types of processing services:

Pre-Built Topologies

Pre-Built topologies provide ready-made frameworks that provide a collection of core data processing operations. The pre-built bridges allow quick configuration, testing and deployment into the data warehouse. The following are the current pre-built data bridges;

  • Deduplication (execute hash checks to identify duplicates, set aside duplicates)
  • Data layout/structure (apply configurable “run-time” schema rules to data in-flight)
  • Format standardization (transform output to valid CSV format as detailed in RFC4180 specifications
  • Routing delivery of data to one or more resource locations (i.e., databases, message brokers, storage, etc.)
  • Data loading (i.e., create tables, load data, etc.)

Custom-Built Topologies

Custom-built topologies are tailored applications that allow the processing of data based on the specific customer requirements of one or more streams of data. This may a custom topology which scores customers based on specific in-store purchase activity combined with enrollment behaviors in a mobile application or website. A custom topology may involve specific processing of data to meet the format and layout requirements to properly load into proprietary internal systems. Custom topologies may also be used to deconstruct your data at the tuple or column level to be able to ensure certain customer data meets regulatory compliance based on the data country of origin or where an individual may reside

Note

A topology, once deployed, is designed to run indefinitely and operate ephemerally.
